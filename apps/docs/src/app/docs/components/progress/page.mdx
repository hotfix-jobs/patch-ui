# Progress

Determinate progress bar for work whose total is knowable: file uploads, multi-step setup, build steps. Four intent variants, three preset heights, plus numeric `height`/`width` overrides for full control.

For short indeterminate waits use `Spinner`; for inline copy use `Spinner variant="dots"`.

## Usage

```tsx
import { Progress } from "@patchui/react";

// Basic
<Progress value={30} />

// Custom max
<Progress value={30} max={40} />

// Intent variants
<Progress value={40} variant="success" />
<Progress value={70} variant="warning" />
<Progress value={90} variant="error" />

// Preset heights
<Progress value={60} size="sm" />   // 4px
<Progress value={60} size="md" />   // 6px (default)
<Progress value={60} size="lg" />   // 10px

// Exact pixel height and/or width
<Progress value={60} height={4} width={200} />
<Progress value={60} height={20} width="50%" />

// Indeterminate (unknown total)
<Progress value={null} />
```

<Callout title="Notes">
- Pair the bar with text that names the work and units: `Uploading 12 of 30 files`, `Building · 1.2 GB / 4 GB`. The bar alone doesn't say what's progressing.
- Pass the real `max` for the ceiling. Hardcoding 100 loses meaning when the ceiling is variable.
- Once the bar fills, swap to a completion state (toast, success row, redirect). Don't leave the filled bar sitting on screen with "Complete" tacked on.
- Indeterminate mode animates a sweeping band. Reserve it for waits with no measurable total. Prefer `Spinner` (ring or dots variant) for sub-second waits.
- Throttle rapid `value` updates in code: updating every animation frame will flood screen readers with `aria-valuenow` changes.
- The fill transitions via CSS `transition-[width]` at overlay duration for a smooth animation without pulling in a motion library.
</Callout>
