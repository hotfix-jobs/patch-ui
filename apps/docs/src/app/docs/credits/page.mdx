# Credits

Patch UI didn't materialize in a vacuum. It builds on years of open-source work from other teams. This page names them.

## Design inspiration

### Vercel Geist

The visual language and system design are inspired by [Vercel's Geist](https://vercel.com/geist). Specifically:

- The `100`→`1000` scale structure with the intent contract at each step (`100`–`300` fills, `400`–`600` borders, `700`–`800` solids, `900`–`1000` text) is Geist's model.
- The palette selection (gray plus blue, red, amber, green, teal, purple, pink) and the four-family typography scale (heading, copy, label, button) follow Geist's structure.
- The two-tone background approach (`background-100` / `background-200`) and the "dark surfaces share the canvas color" convention are directly Geist.
- The semantic status colors (`--error`, `--warning`, `--success`) track Geist's red / amber / blue-800 values directly, per comments in `tokens.css`.
- Several component API surfaces (Calendar, Scroller, ListRow, MiddleTruncate, Breadcrumb) are modeled on Geist's equivalents.

The specific values vary in how close they are to Geist: some (semantic roles above) are direct, others (the accent hue values, the choice of ten scale steps per hue, the palette shortlist) are adaptations, and some (the radius set, the motion timings, the focus ring approach) are independent choices. Patch UI is not a fork or a port, and no React code was copied verbatim. But the system's DNA is Geist's, intentionally. If the library feels familiar, that is why.

### shadcn/ui

The distribution model, copy-in components delivered through a registry rather than an npm package, is Adam Wathan's and [shadcn](https://ui.shadcn.com)'s idea. Patch UI is a shadcn-compatible registry: consumers use the standard `shadcn` CLI to pull components, no custom tooling required. The registry format is theirs.

## Runtime foundations

### Base UI

Every interactive component (Menu, Select, Dialog, Combobox, Autocomplete, Tabs, Accordion, Radio, Switch, Checkbox, Slider, Field, Toast, Tooltip, and more) is built on [Base UI](https://base-ui.com). Base UI handles the hard parts: focus management, keyboard navigation, ARIA relationships, controlled/uncontrolled state, portal rendering. Patch UI puts a visual layer on top.

### Floating UI

Positioning for popovers, menus, selects, and tooltips uses [Floating UI](https://floating-ui.com). Collision detection, flip strategies, and viewport-aware placement are all handled by the underlying primitives Base UI wires up.

### Motion

Overlay animations (Modal, Sheet, Menu open/close) use [Motion](https://motion.dev), the successor to Framer Motion. Motion drives the spring-based enter and exit transitions and respects `prefers-reduced-motion` automatically.

## Styling & utilities

### Tailwind CSS

Patch UI is Tailwind CSS v4 native. The token system is expressed as a `@theme` block, so tokens become first-class utilities (`bg-background-100`, `text-copy-14`) rather than requiring a separate config file. [Tailwind](https://tailwindcss.com) is a direct dependency of the copy-in workflow.

### class-variance-authority

Component variants (`buttonVariants`, `badgeVariants`, and friends) are declared with [cva](https://cva.style). It gives us type-safe variant compositions with a compact syntax.

### clsx + tailwind-merge

The `cn()` helper composes [clsx](https://github.com/lukeed/clsx) and [tailwind-merge](https://github.com/dcastil/tailwind-merge). clsx builds the class string; tailwind-merge dedupes conflicting utilities so `cn("p-4", className)` behaves as you'd expect.

### Lucide

Component demos, docs illustrations, and default icon slots use [Lucide](https://lucide.dev). Consumers can swap in any icon library; Lucide is the one we ship examples with because the outlines match the crisp-minimal visual bar.

## Original work

What is ours:
- The Tailwind v4 three-layer implementation: raw CSS custom properties on `:root`, an `@theme inline` bridge that generates the utility classes, and the compound `.text-heading-*` / `.text-copy-*` / `.text-label-*` / `.text-button-*` classes that apply size + line-height + tracking + weight in one shot.
- The `check:contrast` gate that enforces WCAG minimums in CI so a token change that breaks readability fails the build.
- The recipe layer (`focusRing`, `controlSize`, `colorTransition`) that gives every component the same interaction feel without per-component reinvention.
- The component implementations themselves. All source code is MIT-licensed and lives at [github.com/hotfix-jobs/patch-ui](https://github.com/hotfix-jobs/patch-ui).
- The blocks (AppHeader, Dropzone) as composed patterns you can drop in and modify.

If you fork or copy from Patch UI, this page is a good template. Attribution matters and it's free.
