# Patch UI v2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rewrite Patch UI's design tokens and every component to a 10-step gray + 7 accent scale, tokenized type/spacing/elevation/motion, and a unified component contract — while simplifying each file in the process.

**Architecture:** All tokens live in `packages/react/src/theme/tokens.css` with raw Geist-style names (`--gray-1000`, `--blue-700`, `--copy-14`, `--radius-6`, `--shadow-menu`). Shared behavior lives in `packages/react/src/recipes.ts` (focus ring, control size, state stepping, disabled, color transition). Components compose recipes; component-local class strings are collapsed via `cva`. Two identity choices persist: motion runs 50% faster than the reference system, focus ring stays 1px neutral outline.

**Tech Stack:** Tailwind CSS v4 (`@theme` block bridges CSS custom properties to utility classes), Base UI (`@base-ui/react` primitives), `cva` for variant maps, `clsx + tailwind-merge` via `cn()`, tsup for library build, Next.js 16 + MDX for docs, shadcn registry for distribution.

**Reference spec:** `docs/superpowers/specs/2026-07-01-patch-ui-v2-design.md`

**Rule for every task and commit:** No "vercel" or "geist" strings in any file, path, comment, or commit message except in `apps/docs/src/app/docs/foundations/credits.mdx`.

# Color

Patch UI ships four color layers: two background surfaces, a 10-step gray scale (with a translucent alpha companion), seven accent hues, and three semantic role aliases. Every scale runs 100 to 1000 and encodes intent.

## Backgrounds

| Token | Purpose |
|---|---|
| `--background-100` | Default page and surface fill. |
| `--background-200` | Subtle separation only. Never a general fill. |

## Gray scale

Every step maps to a state, layer, or text tier. The same step index means the same purpose across every hue family.

| Step | Purpose |
|---|---|
| 100 | Default fill |
| 200 | Hover fill |
| 300 | Active fill |
| 400 | Default border |
| 500 | Hover border |
| 600 | Active border |
| 700 | Solid high-contrast fill, disabled text |
| 800 | Solid fill hover |
| 900 | Secondary text and icons |
| 1000 | Primary text and icons |

## Gray alpha

Translucent versions of the gray scale at the same 100–1000 steps. Use these for borders, dividers, and any overlay that must sit on top of a background it doesn't control.

## Accent hues

Seven families, each following the same 100–1000 contract as gray: `blue`, `red`, `amber`, `green`, `teal`, `purple`, `pink`.

Solid accent is reserved for state signals and one hero action per view. Never decorate with color alone — always pair with an icon or text.

## Role aliases

- `--error` → `red-700`
- `--success` → `green-700`
- `--warning` → `amber-700`

## Rules

- Use the gray scale to rank information: `1000` for primary text, `900` for secondary, `700` for disabled.
- Hold WCAG AA (4.5:1) for body text; AAA (7:1) for primary text.
- Never signal state with color alone.
- Never swap `gray-*` for `background-*` — they are separate scales.
- Never use `background-200` as a general fill.
